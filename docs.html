Getting Started

Metraly is a self-hosted engineering metrics platform. This guide walks you from zero to a running instance with demo data in under 5 minutes.

ℹ️
Alpha stage: Metraly is currently in Alpha. APIs may change, but core functionality is production-tested. We welcome bug reports and contributions on GitHub.

Prerequisites

  • Docker 24+ and Docker Compose v2
  • Git
  • 4GB RAM minimum (8GB recommended for demo data seed)
  • Ports 3000 (UI), 8080 (API), 5432 (Postgres), 6379 (Redis) available

Quick Start

1
Clone the repository
bash
$ git clone https://github.com/getmetraly/metraly.git
$ cd metraly
2
Start all services
This starts the Go API server, PostgreSQL 16 + TimescaleDB, and Redis. Migrations run automatically.
bash
$ make docker-up
✓ Starting API server on :8080
✓ PostgreSQL 16 + TimescaleDB ready
✓ Redis ready on :6379
✓ Migrations applied (zero-dependency runner)
3
Seed demo data
Populates the database with 3 months of deterministic demo data across 5 teams.
bash
$ make seed
✓ Seeded 5 teams, 23 repositories
✓ 90 days of DORA metrics
✓ Demo users created (admin/admin)
4
Open the UI
Navigate to http://localhost:3000. Log in with admin / admin.

Installation

Metraly supports Docker Compose for quick setup and Kubernetes/Helm for production deployments.

Docker Compose (Recommended)

docker-compose.yml
version: '3.9'
services:
  api:
    build: ./cmd/api
    ports: ["8080:8080"]
    environment:
      DATABASE_URL: postgres://metraly:metraly@db:5432/metraly
      REDIS_URL: redis://redis:6379
    depends_on: [db, redis]
  db:
    image: timescale/timescaledb:latest-pg16
    volumes: [pgdata:/var/lib/postgresql/data]
  redis:
    image: redis:7-alpine

Makefile Targets

TargetDescription
make docker-upStart all services (API, PostgreSQL, Redis)
make docker-downStop and remove containers
make seedFill DB with deterministic demo data
make migrateRun pending database migrations
make buildBuild the Go binary
make testRun the test suite

Configuration

Metraly is configured entirely via environment variables. No YAML config files to manage.

Core Variables

VariableRequiredDescription
DATABASE_URLYesPostgreSQL connection string (pgx format)
REDIS_URLYesRedis connection string
JWT_PRIVATE_KEYYesRS256 private key in PEM format
JWT_PUBLIC_KEYYesRS256 public key in PEM format
PORTNoAPI port (default: 8080)
LOG_LEVELNodebug | info | warn | error (default: info)

OIDC / SSO

VariableDescription
OIDC_ISSUER_URLIssuer URL from your identity provider (lazy-initialized)
OIDC_CLIENT_IDOAuth2 client ID
OIDC_CLIENT_SECRETOAuth2 client secret
OIDC_REDIRECT_URLCallback URL (e.g. https://metraly.yourdomain.com/auth/oidc/callback)

Generate JWT keys: Run openssl genrsa -out private.pem 4096 and openssl rsa -in private.pem -pubout -out public.pem to generate a key pair.

Architecture
Core Concepts

Architecture

Metraly follows a strict layered architecture with clear separation between HTTP, business logic, and data access.

Stack

LayerTechnologyNotes
API ServerGo (cmd/api/)20 REST endpoints, go-chi/chi/v5 router
Primary DBPostgreSQL 16 + TimescaleDBTime-series metrics, hypertables
Cache / SessionsRedisRefresh tokens, per-operation TTL cache
FrontendReact + TypeScript + ViteFeature-based structure in ui_new/
AuthJWT RS256 + OIDC15-min access tokens, RBAC
LoggingzerologStructured JSON logs

Layered Design

Every feature follows the same pattern: Handler → Biz → Repo → PostgreSQL/Redis

go
// Handler — HTTP binding only
func (h *MetricsHandler) GetMetric(w http.ResponseWriter, r *http.Request) {
    id := chi.URLParam(r, "id")
    result, err := h.biz.GetMetric(r.Context(), id, r.URL.Query())
    if err != nil { h.err(w, err); return }
    h.json(w, result)
}

// Biz — business logic, cache ownership
func (b *MetricsBiz) GetMetric(ctx context.Context, id string, q url.Values) (*Metric, error) {
    if cached := b.cache.Get(id); cached != nil { return cached, nil }
    m, err := b.repo.FindMetric(ctx, id, parseTimeRange(q))
    if err == nil { b.cache.Set(id, m, 5*time.Minute) }
    return m, err
}

Integrations

Integrations

Connect Metraly to your Git providers, CI/CD systems, and project management tools via the Connect Sources wizard or the API.

GitHub

Required scopes: repo, read:org, workflow. Connect via Configure → Connect Sources → GitHub.

GitLab

Works with both GitLab.com and self-hosted instances. Set GITLAB_URL for self-hosted.

Jenkins

⚠️
Jenkins integration is in Beta. Requires Jenkins 2.361+ with the Remote Access API enabled.

Jira

Use a Jira service account API token, not a personal token, for production. Required: project read permissions on boards you want to track.

Alerts & Notifications
Advanced

Alerts & Notifications

Configure threshold-based alerts delivered to Telegram, Slack, or Discord directly from your infrastructure.

🔒
Data privacy: Alert messages are sent directly from your Metraly instance. Only the message content (no raw metrics data) leaves your network.

Alert Rule Syntax

alert rule
# Condition with team filter
cycle_time > 3d  AND  team = "backend"

# With time window
ci_success_rate < 80%  WINDOW 24h

Telegram Setup

1
Create a Telegram Bot
Message @BotFather and use /newbot. Save the bot token.
2
Get the Chat ID
Add the bot to your group, call https://api.telegram.org/bot<TOKEN>/getUpdates.
3
Configure in Metraly
Go to Settings → Notifications → Telegram, paste your bot token and chat ID.

Slack & Discord

Create a Slack Incoming Webhook or Discord server webhook. Paste the URL in Settings → Notifications.

AI Configuration

AI Configuration

Metraly supports three AI modes: built-in provider, bring-your-own API key, and fully air-gapped self-hosted LLM.

🛡️
No training on your data. Metraly never sends your metrics to a training pipeline. AI uses your configured endpoint only, with no telemetry.

Option A: Built-in Provider

.env
AI_ENABLED=true
# No other AI variables needed

Option B: Bring Your Own API Key

.env
AI_ENABLED=true
AI_ENDPOINT=https://api.openai.com/v1
AI_API_KEY=sk-...
AI_MODEL=gpt-4o

Option C: Self-hosted LLM (Air-gapped)

.env (Ollama)
AI_ENABLED=true
AI_ENDPOINT=http://ollama:11434/v1
AI_API_KEY=ollama
AI_MODEL=llama3.1:8b

Contributing

Metraly is built in public. We welcome bug reports, feature ideas, and pull requests.

Development Setup

bash
$ git clone https://github.com/getmetraly/metraly.git && cd metraly
$ make docker-deps    # Start dependencies only
$ go run ./cmd/api/...
$ cd ui_new && npm install && npm run dev

PR Guidelines

  • Follow the Handler → Biz → Repo layered architecture
  • Add unit tests for all Biz layer logic
  • Use zerolog — no fmt.Println
  • All SQL queries use pgx/v5 directly — no ORM
  • Keep PRs focused — one feature or fix per PR

FAQ

Common questions about deploying and using Metraly.

Is Metraly production-ready?

Metraly is in Alpha. Core API, database schema, and auth are stable. Use in production with the understanding that APIs may change between minor versions.

Does Metraly send any telemetry?

No. Zero telemetry, zero callbacks home. You can verify this by auditing the source code or running it in an air-gapped environment.

Can I run Metraly in a fully air-gapped environment?

Yes. Use Option C (self-hosted LLM via Ollama) for AI features. All container images can be mirrored to a private registry.

What does the FSL license mean for me?

FSL allows free use for non-commercial and internal commercial use. After 2 years, each version's license converts to Apache 2.0 automatically. See the Pricing page for details.

How do I upgrade Metraly?

Pull the latest image and run make docker-up. Migrations run automatically. Always back up PostgreSQL before upgrading.
